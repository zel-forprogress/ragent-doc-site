面试高频考点#

RAG 和微调区别#

RAG 适合注入频繁变化的外部知识，更新文档即可；微调适合改变模型能力或风格，但不适合频繁更新事实知识。企业知识库问答一般优先 RAG。

为什么要分块#

整篇文档太大，检索不精准；分块太小，上下文不完整。分块策略要在召回精度和语义完整性之间平衡。

为什么要 Rerank#

向量相似度只能粗筛，Rerank 会重新判断问题和候选 chunk 的相关性，把真正能回答问题的内容排到前面。

为什么要 RocketMQ#

文档解析、分块、Embedding、向量写入耗时较长，不适合放在上传接口同步执行。RocketMQ 可以把入库任务异步化，提高接口响应速度，也便于失败记录和重试。

为什么要 Redis#

Redis 在这个项目里主要用于 Redisson、限流、锁和幂等。例如问答并发限制、文档上传信号量、消息消费幂等等，都适合用 Redis 支撑。

为什么要 MCP#

知识库回答静态文档问题，MCP 调用外部工具解决实时业务问题。比如天气、票务、销售数据，这些不适合只靠向量检索。

为什么要 Trace#

RAG 的错误往往发生在中间步骤，比如改写错、意图错、检索错、Prompt 拼错。Trace 可以记录每个节点输入输出和耗时，帮助定位效果问题。

这个项目最值得讲的亮点#

Ragent 的亮点是工程链路完整，不只是模型调用。它覆盖文档入库、向量检索、意图识别、问题改写、MCP 工具调用、模型路由、SSE 流式输出、Redis 限流、RocketMQ 异步任务和 Trace 排障，能体现完整的企业级 RAG 落地能力。

Ragent 项目总览#

Ragent 是一个 Java 技术栈的企业级 RAG 智能问答系统。它不是只演示“调模型 API”的小 Demo，而是把一个完整 RAG 系统拆成了可落地的工程模块：

  • 用户登录与权限。
  • 会话管理与消息存储。
  • 文档上传、解析、分块、向量化、入库。
  • 知识库管理。
  • 意图树管理。
  • 多通道检索。
  • MCP 工具调用。
  • 大模型路由、降级、流式输出。
  • RAG 链路追踪。
  • 管理后台。

从源码看，Ragent 后端是 Maven 多模块项目：

模块 作用
bootstrap 主业务服务，包含用户、知识库、RAG 问答、入库流水线、管理后台接口
framework 通用能力，例如统一返回、异常、用户上下文、RocketMQ、幂等、Trace
infra-ai 大模型、Embedding、Rerank 的供应商适配与模型路由
mcp-server 独立 MCP 工具服务，提供天气、票务、销售等示例工具
frontend React + Vite 管理后台与问答前端

关键配置文件：

文件 说明
bootstrap/src/main/resources/application.yaml 后端主配置，数据库、Redis、RocketMQ、RustFS、AI Key、模型列表都在这里
mcp-server/src/main/resources/application.yml MCP 服务配置，默认端口 9099
frontend/vite.config.ts 前端开发服务器与 /api 代理配置
resources/database/schema_pg.sql PostgreSQL 表结构与 pgvector 表
resources/database/init_data_pg.sql 初始化管理员账号

RAG 项目经验如何写进简历#

很多简历写“做过 AI 问答系统”，但如果只写“调用大模型接口实现问答”，竞争力不强。Ragent 的优势是可以把 RAG 工程化能力讲出来。

简历写法模板#

可以写成这样：

基于 Java 17 + Spring Boot 3 + React + PostgreSQL/pgvector + Redis + RocketMQ 实现企业级 RAG 智能问答系统，支持知识库管理、文档解析分块、Embedding 向量化、多通道检索、意图树识别、MCP 工具调用、会话记忆、模型路由降级和全链路 Trace。

更细一点：

负责 RAG 问答链路设计与实现：用户问题进入系统后，依次经过会话记忆加载、问题改写与拆分、意图识别、模糊意图澄清、多通道并行检索、结果去重与重排、Prompt 组装、模型流式输出，并通过 Trace 记录每个节点耗时和中间结果，便于后续效果分析和排障。

知识库方向可以写：

负责知识库入库链路：支持文档上传到 S3 兼容对象存储，使用 Apache Tika 抽取文本，提供结构感知分块和固定长度分块策略，调用 Embedding 模型生成向量，写入 PostgreSQL pgvector，并通过 RocketMQ 异步执行分块任务，避免上传接口长时间阻塞。

模型调度方向可以写：

实现多模型路由与容错机制：将聊天、Embedding、Rerank 按能力拆分为不同候选模型，基于优先级、健康状态和深度思考能力选择模型，失败后自动切换下一个候选，提升模型调用稳定性。

面试时怎么展开#

面试官问“你这个 RAG 和普通问答有什么区别”，可以按这条线讲：

  • 普通大模型问答只依赖模型已有知识，容易过时、幻觉、不可控。
  • RAG 会先从企业知识库检索相关内容，再把检索结果作为上下文给模型。
  • 工程上难点不在“调 API”，而在文档入库、分块、检索召回、排序、Prompt、上下文长度、链路追踪和模型容错。
  • Ragent 在这些环节都有代码实现，不只是概念演示。

RAG 基础概念#

什么是 RAG 技术#

RAG 是 Retrieval-Augmented Generation，中文常叫“检索增强生成”。

它解决的问题是：大模型本身不知道你的私有资料，比如公司制度、保险产品规则、内部 API 文档、业务手册。RAG 的做法是：

  • 先把私有文档处理成可检索的知识库。
  • 用户提问时，先检索知识库。
  • 把检索到的内容拼进 Prompt。
  • 再让大模型基于这些内容回答。

一句话理解：

RAG = 先查资料，再让大模型组织答案。

在 Ragent 里，问答链路主要落在：

  • RAGChatServiceImpl
  • StreamChatPipeline
  • QueryRewriteService
  • IntentResolver
  • MultiChannelRetrievalEngine
  • RAGPromptService
  • LLMService

基础篇#

RAG 最小闭环包括四步：

  • 文档入库：把文档解析成文本。
  • 向量化：把文本片段变成 Embedding 向量。
  • 检索：把用户问题也变成向量，找最相似的片段。
  • 生成：把片段作为上下文交给大模型回答。

但真实项目里还要处理：

  • 多轮对话上下文。
  • 用户问题太短、太口语化。
  • 一个问题包含多个子问题。
  • 用户意图不明确。
  • 检索出来的内容重复。
  • 检索结果质量不稳定。
  • 模型接口失败或超时。
  • 前端需要流式输出。

Ragent 的核心价值就在于它把这些工程问题拆开实现了。

向量篇#

向量是 RAG 的检索基础。

普通关键词检索依赖字面匹配，比如“报销流程”和“费用申请流程”可能匹配不上。向量检索会把文本转成语义空间里的数字数组，如果两句话意思接近，向量距离就近。

Ragent 当前默认使用 PostgreSQL + pgvector：

rag:
  vector:
    type: pg

对应实现：

类 作用
PgVectorStoreService pgvector 向量写入和删除
PgRetrieverService pgvector 检索
MilvusVectorStoreService Milvus 版本向量存储实现
MilvusRetrieverService Milvus 检索实现
VectorStoreService 向量存储抽象接口
RetrieverService 检索抽象接口

schema_pg.sql 中的关键表是：

CREATE TABLE t_knowledge_vector (...)
CREATE INDEX idx_kv_embedding ON t_knowledge_vector USING hnsw (embedding vector_cosine_ops);

这表示项目使用 pgvector 的 HNSW 索引做相似度检索。

MCP 篇#

MCP 可以理解为“让模型调用工具的一套协议”。如果用户问的是实时天气、订单查询、票务查询，这些信息不适合只靠知识库检索，而应该调用业务工具。

Ragent 里 MCP 分成两部分：

后端主服务作为 MCP Client：

  • HttpMCPClient
  • MCPClientAutoConfiguration
  • RemoteMCPToolExecutor
  • DefaultMCPToolRegistry

独立 MCP 服务作为 Server：

  • mcp-server/src/main/java/.../MCPEndpoint.java
  • MCPDispatcher.java
  • WeatherMCPExecutor.java
  • TicketMCPExecutor.java
  • SalesMCPExecutor.java

MCP 服务默认端口：

server:
  port: 9099

主服务通过这里连接：

rag:
  mcp:
    servers:
      - name: default
        url: http://localhost:9099

MCP Server 支持三个核心 JSON-RPC 方法：

方法 作用
initialize 初始化协议
tools/list 获取工具列表
tools/call 调用某个工具

对话增强篇#

多轮对话有一个常见问题：用户第二轮可能问“那它怎么申请？”，这里的“它”依赖上一轮上下文。

Ragent 做了几件事：

  • 会话历史加载：ConversationMemoryService
  • 历史存储：JdbcConversationMemoryStore
  • 会话摘要：JdbcConversationMemorySummaryService
  • 问题改写：QueryRewriteService
  • 多问题拆分：MultiQuestionRewriteService

配置项：

rag:
  memory:
    history-keep-turns: 4
    summary-start-turns: 5
    summary-enabled: true
    summary-max-chars: 200

意思是：

  • 保留最近 4 轮对话。
  • 从第 5 轮开始考虑摘要。
  • 摘要控制在较短长度，避免 Prompt 过长。

数据传输协议#

Ragent 前后端问答使用 SSE，也就是 Server-Sent Events。

后端接口：

@GetMapping(value = "/rag/v3/chat", produces = "text/event-stream;charset=UTF-8")

对应文件：

bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/controller/RAGChatController.java

为什么用 SSE：

  • 大模型生成答案是逐字或逐片段返回。
  • 前端可以边接收边渲染。
  • 比普通 HTTP 等完整响应体验更好。

Ragent 的流式输出核心类：

类 作用
StreamCallback 模型流式回调接口
StreamCallbackFactory 创建 SSE 事件处理器
StreamChatEventHandler 把模型输出转成前端事件
SseEmitterSender SSE 发送工具
StreamTaskManager 绑定和取消正在生成的任务

模型检索生成评估与优化#

RAG 效果优化通常围绕三件事：

  • 检索是否找到了正确内容。
  • Prompt 是否把上下文表达清楚。
  • 模型是否按上下文回答，而不是自由发挥。

Ragent 中可以从这些地方观察和优化：

方向 代码或功能
检索通道 VectorGlobalSearchChannel、IntentDirectedSearchChannel
检索后处理 DeduplicationPostProcessor、RerankPostProcessor
Prompt 构建 RAGPromptService、DefaultContextFormatter
模型路由 ModelSelector、ModelRoutingExecutor
链路追踪 RagTraceAspect、RagTraceRecordService
前端 Trace 页面 frontend/src/pages/admin/traces

优化方法：

  • 分块太大：检索噪音多，答案容易泛。
  • 分块太小：上下文断裂，模型看不懂完整语义。
  • topK 太小：召回不足。
  • topK 太大：Prompt 变长，噪音变多。
  • Rerank 缺失：相似度靠前不一定最相关。
  • 意图树不准确：会走错知识库或工具。

面试高频考点#

RAG 和微调有什么区别#

回答思路：

RAG 适合把外部知识实时注入模型，不需要重新训练，更新知识只需要更新知识库。微调适合改变模型行为风格或学习稳定模式，但不适合频繁更新事实知识。企业知识库问答通常优先 RAG。

为什么文档要分块#

回答思路：

因为大模型上下文有限，向量检索也需要较小的语义单元。整篇文档向量化会导致语义过粗，检索不精确；切得太碎又会丢上下文。所以分块策略要平衡召回率和上下文完整性。

为什么要 Rerank#

回答思路：

向量相似度能找语义接近的内容，但不一定最能回答问题。Rerank 会把候选 chunk 和问题重新做相关性判断，把真正有用的内容排到前面，降低 Prompt 噪音。

为什么要多通道检索#

回答思路：

单一向量检索容易漏召回。Ragent 同时支持全局向量检索和意图定向检索，前者覆盖广，后者更精准。多个通道并行执行后再统一去重和排序，可以兼顾召回率和准确性。

为什么要 MCP#

回答思路：

知识库适合回答相对稳定的文档知识，但实时数据或业务系统数据应该通过工具查询。MCP 提供了标准化工具调用方式，让模型可以在需要时调用天气、票务、销售等外部工具。

为什么要异步文档入库#

回答思路：

文档解析、分块、Embedding、向量写入都可能耗时较长。如果放在上传接口里同步执行，用户体验差，也容易超时。Ragent 通过 RocketMQ 把文档分块任务异步化，上传和处理解耦。

为什么要模型路由#

回答思路：

生产环境不能只依赖一个模型供应商。Ragent 把模型按 Chat、Embedding、Rerank 能力配置候选列表，调用失败后自动降级到下一个候选，并记录健康状态，提升系统可用性。

项目最大的亮点怎么讲#

可以这样讲：

这个项目不是简单接一个大模型接口，而是实现了 RAG 的完整工程闭环。文档侧有上传、解析、分块、Embedding、向量入库和异步任务；问答侧有会话记忆、问题改写、意图识别、多通道检索、Rerank、Prompt 构建、MCP 工具调用和 SSE 流式输出；运维排障侧有 Trace、模型路由和限流。
