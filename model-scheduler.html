大模型调度引擎实战#

为什么要模型调度#

真实项目不能把所有请求都绑死到一个模型。

原因：

  • 某个供应商可能超时。
  • 某个模型可能限流。
  • 有些问题需要深度思考模型。
  • Embedding 和 Chat 是不同能力。
  • Rerank 可以有独立模型。

Ragent 的模型调度类#

类 作用
ModelSelector 选候选模型
ModelRoutingExecutor 执行并失败切换
ModelHealthStore 记录模型健康状态
ModelTarget 一次候选目标
RoutingLLMService Chat 路由服务
RoutingEmbeddingService Embedding 路由服务
RoutingRerankService Rerank 路由服务

执行逻辑#

读取配置候选模型
  -> 过滤禁用模型
  -> 深度思考场景过滤 supports-thinking
  -> 按默认模型和 priority 排序
  -> 跳过短时间不可用模型
  -> 调用模型
  -> 成功返回
  -> 失败标记并尝试下一个

面试怎么讲#

我没有在业务代码里直接调用某个固定模型，而是做了模型能力抽象。Chat、Embedding、Rerank 都有独立候选模型列表，由 ModelSelector 根据默认模型、优先级、深度思考能力和健康状态选择候选，再由 ModelRoutingExecutor 执行调用。如果某个模型失败，会记录失败并自动 fallback 到下一个候选。

大模型调度引擎实战#

Ragent 的模型调用不是散落在业务代码中，而是集中到 infra-ai。

模型能力拆分#

项目把模型分成三类：

能力 作用
Chat 生成回答、意图识别、问题改写
Embedding 文本转向量
Rerank 对检索结果重新排序

对应服务：

  • RoutingLLMService
  • RoutingEmbeddingService
  • RoutingRerankService

模型选择#

核心类：

ModelSelector

它会根据：

  • 默认模型。
  • 深度思考模式。
  • 候选模型优先级。
  • 模型是否启用。
  • 模型健康状态。

选出候选列表。

配置示例：

ai:
  chat:
    default-model: qwen3-max
    deep-thinking-model: qwen3-max
    candidates:
      - id: qwen-plus
        provider: bailian
        model: qwen-plus-latest
        priority: 1
      - id: qwen3-max
        provider: bailian
        model: qwen3-max
        supports-thinking: true
        priority: 3

故障转移#

核心类：

ModelRoutingExecutor

执行逻辑：

  • 遍历候选模型。
  • 找到对应 provider 客户端。
  • 判断模型是否允许调用。
  • 调用模型。
  • 成功则返回。
  • 失败则记录失败，尝试下一个模型。
  • 全部失败则抛出远程调用异常。

这个设计适合面试重点讲，因为它体现了工程容错意识。

健康状态#

核心类：

ModelHealthStore

配置：

ai:
  selection:
    failure-threshold: 2
    open-duration-ms: 30000

意思是：某个模型连续失败达到阈值后，会短时间内被跳过，避免每次请求都卡在坏模型上。
