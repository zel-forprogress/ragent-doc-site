A. Ragent#

A.1 这个项目到底是什么#

Ragent 是一个基于 Java 17、Spring Boot 3、React、PostgreSQL/pgvector、Redis、RocketMQ 和对象存储的 RAG 智能问答系统。

一句话理解：

Ragent = 知识库管理 + 文档入库 + 向量检索 + 大模型问答 + MCP 工具调用 + 后台管理 + Trace 排障

很多入门项目只做了这条链路：

用户问题 -> 调大模型 -> 返回答案

Ragent 做的是更完整的工程链路：

文档上传 -> 文档解析 -> 文本分块 -> Embedding -> 向量入库
用户提问 -> 加载历史 -> 问题改写 -> 意图识别 -> 检索/工具调用 -> Prompt 组装 -> 模型流式回答 -> Trace 记录

C. RAG 项目经验如何写进简历#

C.1 简历不要只写“接入大模型”#

写“接入大模型 API 实现智能问答”太薄了，因为这只说明你会调用接口。RAG 项目的简历重点应该是“工程链路”。

你可以按 4 层写：

层次 简历关键词
数据层 文档解析、文本清洗、分块策略、Embedding、向量入库
检索层 pgvector、相似度检索、多通道检索、去重、Rerank
生成层 Prompt 工程、上下文拼接、流式输出、会话记忆
工程层 Redis 限流、RocketMQ 异步任务、S3 对象存储、Trace、模型路由降级

C.2 可直接使用的简历描述#

版本一，适合写项目经历：

基于 Java 17 + Spring Boot 3 + React + PostgreSQL/pgvector + Redis + RocketMQ 实现企业级 RAG 智能问答系统，支持知识库管理、文档解析分块、Embedding 向量化、多通道检索、MCP 工具调用、会话记忆、模型路由降级和全链路 Trace。

版本二，适合强调问答链路：

负责 RAG 问答链路设计：用户问题进入系统后，依次经过会话历史加载、问题改写与拆分、意图树识别、模糊意图澄清、多通道并行检索、结果去重与重排、Prompt 组装、模型流式输出，并通过 Trace 记录每个节点耗时和中间结果。

版本三，适合强调知识库建设：

负责知识库入库链路：使用 RustFS 保存原始文件，基于 Apache Tika 抽取文档文本，支持结构感知分块和固定长度分块，调用 Embedding 模型生成向量，写入 PostgreSQL pgvector，并通过 RocketMQ 异步执行入库任务。

版本四，适合强调稳定性：

设计多模型路由与容错机制，将 Chat、Embedding、Rerank 按能力拆分为候选模型列表，结合优先级、健康状态和深度思考能力选择模型，调用失败后自动切换候选模型，提高模型服务可用性。

C.3 面试时如何讲项目架构#

可以按这个顺序讲：

  • 前端是 React + Vite，负责问答页和管理后台。
  • 后端是 Spring Boot 多模块单体，业务集中在 bootstrap。
  • 数据存储使用 PostgreSQL，向量检索使用 pgvector。
  • Redis 用于限流、锁和 Redisson。
  • RocketMQ 用于文档分块这种耗时任务的异步化。
  • RustFS 用于保存上传文件。
  • infra-ai 屏蔽百炼、SiliconFlow、Ollama 等模型供应商差异。
  • mcp-server 模拟外部业务工具，主服务通过 MCP 协议调用它。

C.4 面试官追问时的回答模板#

问题：你这个项目和普通 ChatGPT 套壳有什么区别？

回答：

普通套壳项目通常只是把用户输入转发给大模型，模型只能基于已有知识回答。Ragent 是完整 RAG 链路，先把企业文档解析、分块、向量化并写入知识库；用户提问时会做会话记忆、问题改写、意图识别和多通道检索，再把检索内容组装进 Prompt 给模型。它还包含 MCP 工具调用、模型路由降级和 Trace 排障，更接近企业应用。

D. RAG 基础概念#

D.1 什么是 RAG 技术？#

RAG 是 Retrieval-Augmented Generation，中文通常叫“检索增强生成”。

你可以用一个生活例子理解：

普通大模型回答 = 闭卷考试
RAG 回答 = 先查资料，再开卷答题

大模型本身不知道你公司的内部制度、产品说明、业务数据。RAG 的核心就是在回答前先检索你的私有知识库，然后把检索结果作为上下文给模型。

完整链路：

离线阶段：文档 -> 解析 -> 分块 -> Embedding -> 向量库
在线阶段：问题 -> 改写 -> Embedding/意图识别 -> 检索 -> Prompt -> LLM -> 答案

Ragent 里的在线问答主线：

RAGChatController
  -> RAGChatServiceImpl
  -> StreamChatPipeline
  -> ConversationMemoryService
  -> QueryRewriteService
  -> IntentResolver
  -> RetrievalEngine
  -> RAGPromptService
  -> LLMService

D.2 RAG 解决了什么问题#

问题 没有 RAG 有 RAG
私有知识 模型不知道公司内部资料 从知识库检索后回答
知识更新 需要重新训练或等模型更新 更新文档和向量即可
幻觉 模型容易编 要求基于检索上下文回答
可解释性 不知道答案依据 可以展示命中的文档片段
成本 长文档全塞 Prompt 很贵 只取相关 chunk

D.3 RAG 不等于简单向量检索#

很多人以为：

RAG = Embedding + 向量数据库 + 大模型

这只是最小 Demo。真实工程至少还要考虑：

  • 文档解析质量。
  • Chunk 粒度。
  • 元数据过滤。
  • 查询改写。
  • 多轮上下文。
  • 多通道召回。
  • Rerank。
  • Prompt 结构。
  • 流式响应。
  • 调用失败降级。
  • Trace 和评估。

Ragent 的代码基本覆盖了这些点，所以它适合拿来学工程化 RAG。

E. RAG 基础概念 - 基础篇#

E.1 什么是大模型#

大模型可以理解成一个“根据上下文预测下一个 token 的语言模型”。它并不是数据库，也不是搜索引擎。它擅长：

  • 理解自然语言。
  • 总结文本。
  • 改写问题。
  • 生成答案。
  • 提取结构化信息。
  • 根据工具返回结果组织表达。

它不擅长：

  • 保证事实永远正确。
  • 记住你的私有文档。
  • 实时查询业务系统。
  • 处理超出上下文长度的大量资料。

所以 RAG 的工程设计，本质是在弥补大模型的这些短板。

E.2 Ragent 中大模型承担哪些任务#

在 Ragent 里，大模型不只是最后回答问题，还参与多个中间环节：

场景 作用 相关类
问题改写 把口语化、依赖上下文的问题改成完整问题 QueryRewriteService
多问题拆分 一个复杂问题拆成多个子问题 MultiQuestionRewriteService
意图识别 判断问题属于哪个业务意图 DefaultIntentClassifier
参数抽取 从用户问题中提取 MCP 工具参数 LLMMCPParameterExtractor
最终回答 基于知识库和工具结果生成答案 LLMService
摘要记忆 压缩长对话历史 ConversationMemorySummaryService

这说明大模型在 RAG 系统里通常是“多个小任务的执行器”，不是只在最后生成答案。

E.3 如何调用大模型#

Ragent 没有在业务代码里直接写 HTTP 调模型，而是放在 infra-ai 模块。

核心接口：

ChatClient
EmbeddingClient
RerankClient

供应商实现：

BaiLianChatClient
SiliconFlowChatClient
OllamaChatClient
SiliconFlowEmbeddingClient
OllamaEmbeddingClient
BaiLianRerankClient
NoopRerankClient

为什么这样设计：

  • 业务代码不用关心百炼、SiliconFlow、Ollama 的接口差异。
  • 后续新增模型供应商，只需要新增客户端实现和配置。
  • 模型调用失败时，可以统一走路由降级。

E.4 如何写好 Prompt 工程#

Prompt 不是随便拼一句“请回答用户问题”。在 RAG 中，Prompt 至少要包含：

  • 角色要求：你是什么助手。
  • 回答边界：只能根据上下文回答，不知道就说不知道。
  • 用户问题：当前问题。
  • 检索上下文：命中的知识片段。
  • MCP 结果：工具返回的数据。
  • 历史上下文：多轮对话记忆。
  • 输出格式：是否需要列表、表格、步骤。

Ragent 的 Prompt 构建主要在：

RAGPromptService
PromptContext
DefaultContextFormatter
PromptTemplateLoader

学习时可以重点观察：

  • StreamChatPipeline.streamLLMResponse 如何构造 PromptContext。
  • RAGPromptService 如何把知识库上下文、MCP 上下文和历史消息组合成 messages。
  • 不同场景下 temperature、topP 如何设置。

例如代码里 MCP 场景会稍微放宽温度：

ctx.hasMcp() ? 0.3D : 0D

这表示：纯知识库回答更追求稳定，工具调用场景需要一点自然表达空间。

E.5 Prompt 工程常见错误#

错误 后果 改法
把所有文档都塞给模型 成本高、噪音大、超上下文 先检索，只放相关 chunk
没有限制回答依据 模型容易自由发挥 明确要求基于上下文
上下文没标来源 不便解释和排障 保留 docId、chunkId、标题
历史对话无限追加 token 越来越多 保留近几轮，旧历史摘要
Prompt 太复杂 模型遵循困难 分任务、多阶段处理

F. RAG 基础概念 - 数据篇#

F.1 用 Apache Tika 解析文档#

企业文档格式很杂：

  • PDF。
  • Word。
  • Excel。
  • PPT。
  • Markdown。
  • TXT。
  • 网页内容。

大模型和向量数据库不能直接理解这些文件格式，所以第一步是“解析成文本”。

Ragent 的解析器结构：

类 作用
DocumentParser 文档解析接口
DocumentParserSelector 根据类型选择解析器
TikaDocumentParser 使用 Apache Tika 解析常见文档
MarkdownDocumentParser 解析 Markdown
ParseResult 解析结果
TextCleanupUtil 文本清洗

代码路径：

bootstrap/src/main/java/com/nageoffer/ai/ragent/core/parser

F.2 为什么需要 Tika#

如果你自己解析 PDF、Word，会遇到很多坑：

  • PDF 可能是扫描件。
  • Word 有标题、表格、页眉页脚。
  • PPT 文本分散在不同元素里。
  • Excel 有多个 sheet。
  • 文件编码不统一。

Apache Tika 的价值是屏蔽大部分文件格式差异，把多种文档转成统一文本。

F.3 Ragent 文档解析流程#

普通分块模式里，核心逻辑在 KnowledgeDocumentServiceImpl.runChunkProcess：

fileStorageService.openStream(documentDO.getFileUrl())
  -> parserSelector.select(ParserType.TIKA.getType()).extractText(...)
  -> chunkingStrategy.chunk(...)
  -> chunkEmbeddingService.embed(...)
  -> persistChunksAndVectorsAtomically(...)

按人话说：

  • 从 RustFS 打开原始文件流。
  • 用 Tika 抽取文本。
  • 按分块策略切成 chunks。
  • 调 Embedding 模型。
  • 同时写业务分块表和向量表。

F.4 数据分块 Chunk 策略与实践#

为什么要分块？

因为整篇文档太大，直接向量化会让语义变粗。比如一篇 20 页制度文档，里面既有请假，又有报销，又有考勤。如果整篇作为一个向量，用户问“年假怎么申请”，检索结果就不够精准。

但分块也不能太小。如果每 20 个字切一块，模型拿到的是碎片，无法理解上下文。

所以分块要平衡：

粒度太大 -> 召回不精准
粒度太小 -> 上下文不完整

Ragent 中的分块接口：

ChunkingStrategy

已实现策略：

策略 类 适用场景
固定长度分块 FixedSizeTextChunker 文本结构不明显、先快速跑通
结构感知分块 StructureAwareTextChunker Markdown、制度文档、标题层级明显的材料

代码路径：

bootstrap/src/main/java/com/nageoffer/ai/ragent/core/chunk

F.5 分块时要保存哪些信息#

一个 chunk 不应该只有 content，还应该有元数据。

常见元数据：

元数据 作用
kbId 属于哪个知识库
docId 属于哪个文档
chunkId 片段唯一 ID
chunkIndex 在文档里的顺序
contentHash 判断内容是否重复
enabled 是否参与检索
fileType 文件类型
docName 文档名称

Ragent 的 chunk 表：

t_knowledge_chunk

向量表：

t_knowledge_vector

F.6 元数据的作用与管理#

元数据在 RAG 中非常重要。它不是“附属信息”，而是检索过滤和结果解释的基础。

典型作用：

  • 权限过滤：用户只能检索自己有权限的知识库。
  • 范围限定：只在某个知识库或某类文档里查。
  • 答案溯源：告诉用户答案来自哪个文档。
  • 增量更新：删除某个文档时，只删除它的向量。
  • 检索调优：按文档类型、业务域、时间等做过滤。

Ragent 在删除文档时会同时删除向量：

vectorStoreService.deleteDocumentVectors(collectionName, docId)

这就是元数据中 docId 的价值。

F.7 数据篇实践任务#

G. RAG 基础概念 - 向量篇#

G.1 从文本到向量：理解 Embedding#

Embedding 是把文本转换成一串数字。

例如：

"请假流程" -> [0.12, -0.04, 0.88, ...]
"年假怎么申请" -> [0.10, -0.02, 0.83, ...]

如果两句话语义接近，它们的向量距离也接近。

所以向量检索不是按字面匹配，而是按语义相似度匹配。

G.2 为什么不用普通 SQL like#

普通 SQL：

where content like '%请假%'

只能匹配“请假”这个词。如果用户问：

年休假怎么提？

文档写的是：

员工年假申请流程如下

关键词可能匹配不好。Embedding 可以把“年休假”“年假”“休假申请”映射到相近语义空间。

G.3 Ragent 如何生成 Embedding#

核心类：

类 作用
ChunkEmbeddingService 给 chunk 批量补充向量
EmbeddingService Embedding 服务接口
RoutingEmbeddingService 带路由能力的 Embedding 服务
SiliconFlowEmbeddingClient SiliconFlow Embedding 客户端
OllamaEmbeddingClient Ollama Embedding 客户端

配置：

ai:
  embedding:
    default-model: qwen-emb-8b
    candidates:
      - id: qwen-emb-8b
        provider: siliconflow
        model: Qwen/Qwen3-Embedding-8B
        dimension: ${rag.default.dimension}

默认维度：

rag:
  default:
    dimension: 1536

维度必须一致。模型输出 1536 维，数据库 vector 字段也要按 1536 维设计。

G.4 向量数据库的原理与选型#

向量数据库解决的是：

给我一个查询向量，请快速找出最相似的 N 个向量。

常见相似度：

指标 含义
Cosine 余弦相似度，关注方向
L2 欧氏距离，关注空间距离
Dot Product 点积，常用于归一化向量

Ragent 配置：

rag:
  default:
    metric-type: COSINE

G.5 pgvector 与 Milvus 怎么选#

Ragent 同时保留了 pgvector 和 Milvus 实现。

方案 优点 缺点 适用
PostgreSQL + pgvector 部署简单，业务数据和向量数据在一个库里 大规模向量检索能力不如专业向量库 学习、原型、中小规模
Milvus 专业向量数据库，适合大规模向量检索 部署复杂，需要额外运维 大规模生产场景

当前项目默认：

rag:
  vector:
    type: pg

所以首次学习建议先用 pgvector，不要一开始就部署 Milvus。

I. RAG 基础概念 - MCP 篇#

I.1 MCP 是什么#

MCP 可以理解成大模型调用外部工具的一套标准协议。

RAG 适合回答文档知识，但有些问题不该从知识库里找：

K.1 为什么问答用 SSE#

大模型生成答案需要时间。如果等完整答案生成后再返回，用户会感觉卡住。

SSE 可以让后端不断把片段推给前端：

模型生成一点 -> 后端发送一点 -> 前端渲染一点

后端接口：

控制器：

RAGChatController

输出类型：

text/event-stream;charset=UTF-8

K.2 SSE 和 WebSocket 的区别#

对比 SSE WebSocket
通信方向 服务端到客户端为主 双向
协议复杂度 简单 较复杂
断线重连 浏览器原生支持 需要自己处理
适合场景 大模型流式输出、通知 实时协作、游戏、双向聊天

Ragent 的问答流是服务端持续推送，所以 SSE 足够。

R. 大模型调度引擎实战#

R.1 为什么要模型调度#

真实项目不能把所有请求都绑死到一个模型。

原因：

  • 某个供应商可能超时。
  • 某个模型可能限流。
  • 有些问题需要深度思考模型。
  • Embedding 和 Chat 是不同能力。
  • Rerank 可以有独立模型。

R.2 Ragent 的模型调度类#

类 作用
ModelSelector 选候选模型
ModelRoutingExecutor 执行并失败切换
ModelHealthStore 记录模型健康状态
ModelTarget 一次候选目标
RoutingLLMService Chat 路由服务
RoutingEmbeddingService Embedding 路由服务
RoutingRerankService Rerank 路由服务

R.3 执行逻辑#

读取配置候选模型
  -> 过滤禁用模型
  -> 深度思考场景过滤 supports-thinking
  -> 按默认模型和 priority 排序
  -> 跳过短时间不可用模型
  -> 调用模型
  -> 成功返回
  -> 失败标记并尝试下一个

R.4 面试怎么讲#

我没有在业务代码里直接调用某个固定模型，而是做了模型能力抽象。Chat、Embedding、Rerank 都有独立候选模型列表，由 ModelSelector 根据默认模型、优先级、深度思考能力和健康状态选择候选，再由 ModelRoutingExecutor 执行调用。如果某个模型失败，会记录失败并自动 fallback 到下一个候选。

T. 面试高频考点#

T.1 RAG 和微调区别#

RAG 适合注入频繁变化的外部知识，更新文档即可；微调适合改变模型能力或风格，但不适合频繁更新事实知识。企业知识库问答一般优先 RAG。

T.2 为什么要分块#

整篇文档太大，检索不精准；分块太小，上下文不完整。分块策略要在召回精度和语义完整性之间平衡。

T.3 为什么要 Rerank#

向量相似度只能粗筛，Rerank 会重新判断问题和候选 chunk 的相关性，把真正能回答问题的内容排到前面。

T.4 为什么要 RocketMQ#

文档解析、分块、Embedding、向量写入耗时较长，不适合放在上传接口同步执行。RocketMQ 可以把入库任务异步化，提高接口响应速度，也便于失败记录和重试。

T.5 为什么要 Redis#

Redis 在这个项目里主要用于 Redisson、限流、锁和幂等。例如问答并发限制、文档上传信号量、消息消费幂等等，都适合用 Redis 支撑。

T.6 为什么要 MCP#

知识库回答静态文档问题，MCP 调用外部工具解决实时业务问题。比如天气、票务、销售数据，这些不适合只靠向量检索。

T.7 为什么要 Trace#

RAG 的错误往往发生在中间步骤，比如改写错、意图错、检索错、Prompt 拼错。Trace 可以记录每个节点输入输出和耗时，帮助定位效果问题。

T.8 这个项目最值得讲的亮点#

Ragent 的亮点是工程链路完整，不只是模型调用。它覆盖文档入库、向量检索、意图识别、问题改写、MCP 工具调用、模型路由、SSE 流式输出、Redis 限流、RocketMQ 异步任务和 Trace 排障，能体现完整的企业级 RAG 落地能力。

4. RAG 基础概念#

4.1 什么是 RAG 技术#

RAG 是 Retrieval-Augmented Generation，中文常叫“检索增强生成”。

它解决的问题是：大模型本身不知道你的私有资料，比如公司制度、保险产品规则、内部 API 文档、业务手册。RAG 的做法是：

  • 先把私有文档处理成可检索的知识库。
  • 用户提问时，先检索知识库。
  • 把检索到的内容拼进 Prompt。
  • 再让大模型基于这些内容回答。

一句话理解：

RAG = 先查资料，再让大模型组织答案。

在 Ragent 里，问答链路主要落在：

  • RAGChatServiceImpl
  • StreamChatPipeline
  • QueryRewriteService
  • IntentResolver
  • MultiChannelRetrievalEngine
  • RAGPromptService
  • LLMService

4.2 基础篇#

RAG 最小闭环包括四步：

  • 文档入库：把文档解析成文本。
  • 向量化：把文本片段变成 Embedding 向量。
  • 检索：把用户问题也变成向量，找最相似的片段。
  • 生成：把片段作为上下文交给大模型回答。

但真实项目里还要处理：

  • 多轮对话上下文。
  • 用户问题太短、太口语化。
  • 一个问题包含多个子问题。
  • 用户意图不明确。
  • 检索出来的内容重复。
  • 检索结果质量不稳定。
  • 模型接口失败或超时。
  • 前端需要流式输出。

Ragent 的核心价值就在于它把这些工程问题拆开实现了。

4.3 数据篇#

4.7 对话增强篇#

多轮对话有一个常见问题：用户第二轮可能问“那它怎么申请？”，这里的“它”依赖上一轮上下文。

Ragent 做了几件事：

  • 会话历史加载：ConversationMemoryService
  • 历史存储：JdbcConversationMemoryStore
  • 会话摘要：JdbcConversationMemorySummaryService
  • 问题改写：QueryRewriteService
  • 多问题拆分：MultiQuestionRewriteService

配置项：

rag:
  memory:
    history-keep-turns: 4
    summary-start-turns: 5
    summary-enabled: true
    summary-max-chars: 200

意思是：

  • 保留最近 4 轮对话。
  • 从第 5 轮开始考虑摘要。
  • 摘要控制在较短长度，避免 Prompt 过长。

  • 前端可以边接收边渲染。

4.9 模型检索生成评估与优化#

RAG 效果优化通常围绕三件事：

  • 检索是否找到了正确内容。
  • Prompt 是否把上下文表达清楚。
  • 模型是否按上下文回答，而不是自由发挥。

Ragent 中可以从这些地方观察和优化：

方向 代码或功能
检索通道 VectorGlobalSearchChannel、IntentDirectedSearchChannel
检索后处理 DeduplicationPostProcessor、RerankPostProcessor
Prompt 构建 RAGPromptService、DefaultContextFormatter
模型路由 ModelSelector、ModelRoutingExecutor
链路追踪 RagTraceAspect、RagTraceRecordService
前端 Trace 页面 frontend/src/pages/admin/traces

优化方法：

  • 分块太大：检索噪音多，答案容易泛。
  • 分块太小：上下文断裂，模型看不懂完整语义。
  • topK 太小：召回不足。
  • topK 太大：Prompt 变长，噪音变多。
  • Rerank 缺失：相似度靠前不一定最相关。
  • 意图树不准确：会走错知识库或工具。

10. 大模型调度引擎实战#

Ragent 的模型调用不是散落在业务代码中，而是集中到 infra-ai。

10.1 模型能力拆分#

项目把模型分成三类：

能力 作用
Chat 生成回答、意图识别、问题改写

对应服务：

  • RoutingLLMService

10.2 模型选择#

核心类：

ModelSelector

它会根据：

  • 默认模型。
  • 深度思考模式。
  • 候选模型优先级。
  • 模型是否启用。
  • 模型健康状态。

选出候选列表。

配置示例：

ai:
  chat:
    default-model: qwen3-max
    deep-thinking-model: qwen3-max
    candidates:
      - id: qwen-plus
        provider: bailian
        model: qwen-plus-latest
        priority: 1
      - id: qwen3-max
        provider: bailian
        model: qwen3-max
        supports-thinking: true
        priority: 3

10.3 故障转移#

核心类：

ModelRoutingExecutor

执行逻辑：

  • 遍历候选模型。
  • 找到对应 provider 客户端。
  • 判断模型是否允许调用。
  • 调用模型。
  • 成功则返回。
  • 失败则记录失败，尝试下一个模型。
  • 全部失败则抛出远程调用异常。

这个设计适合面试重点讲，因为它体现了工程容错意识。

10.4 健康状态#

核心类：

ModelHealthStore

配置：

ai:
  selection:
    failure-threshold: 2
    open-duration-ms: 30000

意思是：某个模型连续失败达到阈值后，会短时间内被跳过，避免每次请求都卡在坏模型上。

12. 面试高频考点#

12.1 RAG 和微调有什么区别#

回答思路：

RAG 适合把外部知识实时注入模型，不需要重新训练，更新知识只需要更新知识库。微调适合改变模型行为风格或学习稳定模式，但不适合频繁更新事实知识。企业知识库问答通常优先 RAG。

12.2 为什么文档要分块#

回答思路：

因为大模型上下文有限，向量检索也需要较小的语义单元。整篇文档向量化会导致语义过粗，检索不精确；切得太碎又会丢上下文。所以分块策略要平衡召回率和上下文完整性。

12.3 为什么要 Rerank#

回答思路：

向量相似度能找语义接近的内容，但不一定最能回答问题。Rerank 会把候选 chunk 和问题重新做相关性判断，把真正有用的内容排到前面，降低 Prompt 噪音。

12.4 为什么要多通道检索#

回答思路：

单一向量检索容易漏召回。Ragent 同时支持全局向量检索和意图定向检索，前者覆盖广，后者更精准。多个通道并行执行后再统一去重和排序，可以兼顾召回率和准确性。

12.5 为什么要 MCP#

回答思路：

知识库适合回答相对稳定的文档知识，但实时数据或业务系统数据应该通过工具查询。MCP 提供了标准化工具调用方式，让模型可以在需要时调用天气、票务、销售等外部工具。

12.6 为什么要异步文档入库#

回答思路：

文档解析、分块、Embedding、向量写入都可能耗时较长。如果放在上传接口里同步执行，用户体验差，也容易超时。Ragent 通过 RocketMQ 把文档分块任务异步化，上传和处理解耦。

12.7 为什么要模型路由#

回答思路：

生产环境不能只依赖一个模型供应商。Ragent 把模型按 Chat、Embedding、Rerank 能力配置候选列表，调用失败后自动降级到下一个候选，并记录健康状态，提升系统可用性。

12.8 项目最大的亮点怎么讲#

可以这样讲：

这个项目不是简单接一个大模型接口，而是实现了 RAG 的完整工程闭环。文档侧有上传、解析、分块、Embedding、向量入库和异步任务；问答侧有会话记忆、问题改写、意图识别、多通道检索、Rerank、Prompt 构建、MCP 工具调用和 SSE 流式输出；运维排障侧有 Trace、模型路由和限流。

14. 推荐学习路线#

如果你是第一次学这个项目，建议这样读：

  • 先跑起来：中间件、数据库、MCP、后端、前端。
  • 登录后台：看知识库、意图树、Trace 页面。
  • 看数据库：理解每张表存什么。
  • 看文档上传：从 KnowledgeDocumentController 读到 KnowledgeDocumentServiceImpl。
  • 看问答链路：从 RAGChatController 读到 StreamChatPipeline。
  • 看检索链路：读 MultiChannelRetrievalEngine 和两个 SearchChannel。
  • 看模型路由：读 ModelSelector 和 ModelRoutingExecutor。
  • 看 MCP：读 mcp-server 和主服务里的 HttpMCPClient。
  • 看 Trace：理解如何排查 RAG 效果问题。
  • 最后整理成简历和面试话术。

15. 一句话总结#

Ragent 的价值不在于“接入了大模型”，而在于它用 Java 工程把 RAG 系统真正拆成了可维护的链路：文档入库、向量检索、意图识别、工具调用、模型路由、流式响应和链路追踪。你学这个项目，重点不是背概念，而是理解每个概念在代码里怎么落地。
