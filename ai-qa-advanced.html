AI 知识问答篇#

多通道检索#

核心类：

MultiChannelRetrievalEngine

执行过程：

找到启用的 SearchChannel
  -> 并行执行
  -> 收集结果
  -> 执行后处理器链
  -> 返回最终 chunks

已实现通道：

通道 类
全局向量检索 VectorGlobalSearchChannel
意图定向检索 IntentDirectedSearchChannel

后处理：

后处理 类
去重 DeduplicationPostProcessor
重排 RerankPostProcessor

为什么要意图定向检索#

如果所有问题都全局搜，容易搜到无关文档。意图树可以先判断问题属于哪个业务域，再缩小检索范围。

例如：

用户问“密码怎么重置”

意图树可以定位到：

IT 支持 -> 账号权限

这样比全库检索更稳定。

空检索怎么处理#

Ragent 在 handleEmptyRetrieval 中处理空结果。

它不会让模型硬编，而是返回：

未检索到与问题相关的文档内容

这是 RAG 系统很重要的边界意识。

AI 知识问答篇#

一次问答从前端到后端#

前端：

  • ChatInput.tsx 收集输入。
  • chatService.ts 发起请求。
  • useStreamResponse.ts 处理 SSE。
  • MessageList.tsx 渲染消息。

后端：

  • RAGChatController 接收请求。
  • RAGChatServiceImpl 创建会话 ID、任务 ID、回调。
  • StreamChatPipeline 执行完整链路。

为什么要意图树#

如果所有问题都做全局向量检索，会出现两个问题：

  • 检索范围太大，容易找错。
  • 不知道什么时候该调用工具。

意图树的作用是给问题先分类。例如：

企业制度 -> 人事制度 -> 请假流程
企业制度 -> IT 支持 -> 账号权限
业务工具 -> 天气查询
业务工具 -> 票务查询

Ragent 的意图节点表：

t_intent_node

前端管理页面：

frontend/src/pages/admin/intent-tree

为什么要查询词映射#

用户说法和文档说法不一定一致：

用户说法 文档说法
报销 费用报销
入职 员工入职流程
电脑坏了 IT 设备故障

查询词映射可以把口语词转换成更适合检索的业务词。

核心类：

  • QueryTermMappingService
  • QueryTermMappingCacheManager
  • QueryTermMappingAdminService

数据表：

t_query_term_mapping

为什么要 Trace#

RAG 出错时，不能只看最终答案。你要知道：

  • 问题改写成了什么。
  • 识别到了什么意图。
  • 检索到了哪些 chunk。
  • Rerank 后排序如何。
  • Prompt 最后组装了什么。
  • 模型调用是否失败。

Ragent 用 Trace 记录这些信息。

核心类：

  • RagTraceAspect
  • RagTraceContext
  • RagTraceRecordService
  • RagTraceQueryService

数据表：

  • t_rag_trace_run
  • t_rag_trace_node

前端页面：

frontend/src/pages/admin/traces
