RAG 基础概念#

什么是 RAG 技术？#

RAG 是 Retrieval-Augmented Generation，中文通常叫“检索增强生成”。

你可以用一个生活例子理解：

普通大模型回答 = 闭卷考试
RAG 回答 = 先查资料，再开卷答题

大模型本身不知道你公司的内部制度、产品说明、业务数据。RAG 的核心就是在回答前先检索你的私有知识库，然后把检索结果作为上下文给模型。

完整链路：

离线阶段：文档 -> 解析 -> 分块 -> Embedding -> 向量库
在线阶段：问题 -> 改写 -> Embedding/意图识别 -> 检索 -> Prompt -> LLM -> 答案

Ragent 里的在线问答主线：

RAGChatController
  -> RAGChatServiceImpl
  -> StreamChatPipeline
  -> ConversationMemoryService
  -> QueryRewriteService
  -> IntentResolver
  -> RetrievalEngine
  -> RAGPromptService
  -> LLMService

RAG 解决了什么问题#

问题 没有 RAG 有 RAG
私有知识 模型不知道公司内部资料 从知识库检索后回答
知识更新 需要重新训练或等模型更新 更新文档和向量即可
幻觉 模型容易编 要求基于检索上下文回答
可解释性 不知道答案依据 可以展示命中的文档片段
成本 长文档全塞 Prompt 很贵 只取相关 chunk

RAG 不等于简单向量检索#

很多人以为：

RAG = Embedding + 向量数据库 + 大模型

这只是最小 Demo。真实工程至少还要考虑：

  • 文档解析质量。
  • Chunk 粒度。
  • 元数据过滤。
  • 查询改写。
  • 多轮上下文。
  • 多通道召回。
  • Rerank。
  • Prompt 结构。
  • 流式响应。
  • 调用失败降级。
  • Trace 和评估。

Ragent 的代码基本覆盖了这些点，所以它适合拿来学工程化 RAG。

RAG 基础概念 - 基础篇#

什么是大模型#

大模型可以理解成一个“根据上下文预测下一个 token 的语言模型”。它并不是数据库，也不是搜索引擎。它擅长：

  • 理解自然语言。
  • 总结文本。
  • 改写问题。
  • 生成答案。
  • 提取结构化信息。
  • 根据工具返回结果组织表达。

它不擅长：

  • 保证事实永远正确。
  • 记住你的私有文档。
  • 实时查询业务系统。
  • 处理超出上下文长度的大量资料。

所以 RAG 的工程设计，本质是在弥补大模型的这些短板。

Ragent 中大模型承担哪些任务#

在 Ragent 里，大模型不只是最后回答问题，还参与多个中间环节：

场景 作用 相关类
问题改写 把口语化、依赖上下文的问题改成完整问题 QueryRewriteService
多问题拆分 一个复杂问题拆成多个子问题 MultiQuestionRewriteService
意图识别 判断问题属于哪个业务意图 DefaultIntentClassifier
参数抽取 从用户问题中提取 MCP 工具参数 LLMMCPParameterExtractor
最终回答 基于知识库和工具结果生成答案 LLMService
摘要记忆 压缩长对话历史 ConversationMemorySummaryService

这说明大模型在 RAG 系统里通常是“多个小任务的执行器”，不是只在最后生成答案。

如何调用大模型#

Ragent 没有在业务代码里直接写 HTTP 调模型，而是放在 infra-ai 模块。

核心接口：

ChatClient
EmbeddingClient
RerankClient

供应商实现：

BaiLianChatClient
SiliconFlowChatClient
OllamaChatClient
SiliconFlowEmbeddingClient
OllamaEmbeddingClient
BaiLianRerankClient
NoopRerankClient

为什么这样设计：

  • 业务代码不用关心百炼、SiliconFlow、Ollama 的接口差异。
  • 后续新增模型供应商，只需要新增客户端实现和配置。
  • 模型调用失败时，可以统一走路由降级。

如何写好 Prompt 工程#

Prompt 不是随便拼一句“请回答用户问题”。在 RAG 中，Prompt 至少要包含：

  • 角色要求：你是什么助手。
  • 回答边界：只能根据上下文回答，不知道就说不知道。
  • 用户问题：当前问题。
  • 检索上下文：命中的知识片段。
  • MCP 结果：工具返回的数据。
  • 历史上下文：多轮对话记忆。
  • 输出格式：是否需要列表、表格、步骤。

Ragent 的 Prompt 构建主要在：

RAGPromptService
PromptContext
DefaultContextFormatter
PromptTemplateLoader

学习时可以重点观察：

  • StreamChatPipeline.streamLLMResponse 如何构造 PromptContext。
  • RAGPromptService 如何把知识库上下文、MCP 上下文和历史消息组合成 messages。
  • 不同场景下 temperature、topP 如何设置。

例如代码里 MCP 场景会稍微放宽温度：

ctx.hasMcp() ? 0.3D : 0D

这表示：纯知识库回答更追求稳定，工具调用场景需要一点自然表达空间。

Prompt 工程常见错误#

错误 后果 改法
把所有文档都塞给模型 成本高、噪音大、超上下文 先检索，只放相关 chunk
没有限制回答依据 模型容易自由发挥 明确要求基于上下文
上下文没标来源 不便解释和排障 保留 docId、chunkId、标题
历史对话无限追加 token 越来越多 保留近几轮，旧历史摘要
Prompt 太复杂 模型遵循困难 分任务、多阶段处理

RAG 基础概念 - 数据篇#

用 Apache Tika 解析文档#

企业文档格式很杂：

  • PDF。
  • Word。
  • Excel。
  • PPT。
  • Markdown。
  • TXT。
  • 网页内容。

大模型和向量数据库不能直接理解这些文件格式，所以第一步是“解析成文本”。

Ragent 的解析器结构：

类 作用
DocumentParser 文档解析接口
DocumentParserSelector 根据类型选择解析器
TikaDocumentParser 使用 Apache Tika 解析常见文档
MarkdownDocumentParser 解析 Markdown
ParseResult 解析结果
TextCleanupUtil 文本清洗

代码路径：

bootstrap/src/main/java/com/nageoffer/ai/ragent/core/parser

为什么需要 Tika#

如果你自己解析 PDF、Word，会遇到很多坑：

  • PDF 可能是扫描件。
  • Word 有标题、表格、页眉页脚。
  • PPT 文本分散在不同元素里。
  • Excel 有多个 sheet。
  • 文件编码不统一。

Apache Tika 的价值是屏蔽大部分文件格式差异，把多种文档转成统一文本。

Ragent 文档解析流程#

普通分块模式里，核心逻辑在 KnowledgeDocumentServiceImpl.runChunkProcess：

fileStorageService.openStream(documentDO.getFileUrl())
  -> parserSelector.select(ParserType.TIKA.getType()).extractText(...)
  -> chunkingStrategy.chunk(...)
  -> chunkEmbeddingService.embed(...)
  -> persistChunksAndVectorsAtomically(...)

按人话说：

  • 从 RustFS 打开原始文件流。
  • 用 Tika 抽取文本。
  • 按分块策略切成 chunks。
  • 调 Embedding 模型。
  • 同时写业务分块表和向量表。

数据分块 Chunk 策略与实践#

为什么要分块？

因为整篇文档太大，直接向量化会让语义变粗。比如一篇 20 页制度文档，里面既有请假，又有报销，又有考勤。如果整篇作为一个向量，用户问“年假怎么申请”，检索结果就不够精准。

但分块也不能太小。如果每 20 个字切一块，模型拿到的是碎片，无法理解上下文。

所以分块要平衡：

粒度太大 -> 召回不精准
粒度太小 -> 上下文不完整

Ragent 中的分块接口：

ChunkingStrategy

已实现策略：

策略 类 适用场景
固定长度分块 FixedSizeTextChunker 文本结构不明显、先快速跑通
结构感知分块 StructureAwareTextChunker Markdown、制度文档、标题层级明显的材料

代码路径：

bootstrap/src/main/java/com/nageoffer/ai/ragent/core/chunk

分块时要保存哪些信息#

一个 chunk 不应该只有 content，还应该有元数据。

常见元数据：

元数据 作用
kbId 属于哪个知识库
docId 属于哪个文档
chunkId 片段唯一 ID
chunkIndex 在文档里的顺序
contentHash 判断内容是否重复
enabled 是否参与检索
fileType 文件类型
docName 文档名称

Ragent 的 chunk 表：

t_knowledge_chunk

向量表：

t_knowledge_vector

元数据的作用与管理#

元数据在 RAG 中非常重要。它不是“附属信息”，而是检索过滤和结果解释的基础。

典型作用：

  • 权限过滤：用户只能检索自己有权限的知识库。
  • 范围限定：只在某个知识库或某类文档里查。
  • 答案溯源：告诉用户答案来自哪个文档。
  • 增量更新：删除某个文档时，只删除它的向量。
  • 检索调优：按文档类型、业务域、时间等做过滤。

Ragent 在删除文档时会同时删除向量：

vectorStoreService.deleteDocumentVectors(collectionName, docId)

这就是元数据中 docId 的价值。

数据篇实践任务#

你可以用下面的方式练习：

  • 启动项目并登录后台。
  • 创建一个知识库。
  • 上传一份 Markdown 或 PDF。
  • 点击开始分块。
  • 查看 t_knowledge_document 状态变化。
  • 查看 t_knowledge_chunk 是否生成分块。
  • 查看 t_knowledge_vector 是否生成向量。
  • 去 Trace 页面观察问答是否命中文档片段。

常用 SQL：

select id, name, collection_name from t_knowledge_base;
select id, doc_name, status, chunk_count from t_knowledge_document;
select doc_id, chunk_index, left(content, 80) from t_knowledge_chunk limit 10;
select collection_name, doc_id, chunk_id from t_knowledge_vector limit 10;

RAG 基础概念 - 向量篇#

从文本到向量：理解 Embedding#

Embedding 是把文本转换成一串数字。

例如：

"请假流程" -> [0.12, -0.04, 0.88, ...]
"年假怎么申请" -> [0.10, -0.02, 0.83, ...]

如果两句话语义接近，它们的向量距离也接近。

所以向量检索不是按字面匹配，而是按语义相似度匹配。

为什么不用普通 SQL like#

普通 SQL：

where content like '%请假%'

只能匹配“请假”这个词。如果用户问：

年休假怎么提？

文档写的是：

员工年假申请流程如下

关键词可能匹配不好。Embedding 可以把“年休假”“年假”“休假申请”映射到相近语义空间。

Ragent 如何生成 Embedding#

核心类：

类 作用
ChunkEmbeddingService 给 chunk 批量补充向量
EmbeddingService Embedding 服务接口
RoutingEmbeddingService 带路由能力的 Embedding 服务
SiliconFlowEmbeddingClient SiliconFlow Embedding 客户端
OllamaEmbeddingClient Ollama Embedding 客户端

配置：

ai:
  embedding:
    default-model: qwen-emb-8b
    candidates:
      - id: qwen-emb-8b
        provider: siliconflow
        model: Qwen/Qwen3-Embedding-8B
        dimension: ${rag.default.dimension}

默认维度：

rag:
  default:
    dimension: 1536

维度必须一致。模型输出 1536 维，数据库 vector 字段也要按 1536 维设计。

向量数据库的原理与选型#

向量数据库解决的是：

给我一个查询向量，请快速找出最相似的 N 个向量。

常见相似度：

指标 含义
Cosine 余弦相似度，关注方向
L2 欧氏距离，关注空间距离
Dot Product 点积，常用于归一化向量

Ragent 配置：

rag:
  default:
    metric-type: COSINE

pgvector 与 Milvus 怎么选#

Ragent 同时保留了 pgvector 和 Milvus 实现。

方案 优点 缺点 适用
PostgreSQL + pgvector 部署简单，业务数据和向量数据在一个库里 大规模向量检索能力不如专业向量库 学习、原型、中小规模
Milvus 专业向量数据库，适合大规模向量检索 部署复杂，需要额外运维 大规模生产场景

当前项目默认：

rag:
  vector:
    type: pg

所以首次学习建议先用 pgvector，不要一开始就部署 Milvus。

Ragent 向量检索实现#

相关类：

类 说明
VectorStoreService 向量写入、删除抽象
PgVectorStoreService PostgreSQL 向量写入
MilvusVectorStoreService Milvus 向量写入
RetrieverService 检索抽象
PgRetrieverService PostgreSQL 检索
MilvusRetrieverService Milvus 检索

数据库脚本里有 HNSW 索引：

CREATE INDEX idx_kv_embedding ON t_knowledge_vector USING hnsw (embedding vector_cosine_ops);

HNSW 是常见近似最近邻索引，适合快速找相似向量。

向量篇实践任务#

  • 上传一份文档并分块。

  • 观察旧向量是否被删除、新向量是否写入。

RAG 基础概念 - MCP 篇#

MCP 是什么#

MCP 可以理解成大模型调用外部工具的一套标准协议。

RAG 适合回答文档知识，但有些问题不该从知识库里找：

今天上海天气怎么样？
帮我查一下订单销量。
查询某车次余票。

这些问题需要实时调用业务系统。MCP 就是让模型能“看见并调用工具”。

Ragent 中 MCP 的结构#

Ragent 有两个角色：

角色 模块 作用
MCP Client bootstrap 主问答服务，发现工具并调用工具
MCP Server mcp-server 提供工具列表和工具调用接口

MCP Server 入口：

mcp-server/src/main/java/com/nageoffer/ai/ragent/mcp/endpoint/MCPEndpoint.java

分发器：

MCPDispatcher

支持方法：

JSON-RPC 方法 作用
initialize 初始化
tools/list 返回工具列表
tools/call 调用工具

示例工具#

工具 类 作用
weather_query WeatherMCPExecutor 天气查询
ticket_query TicketMCPExecutor 票务查询
sales_query SalesMCPExecutor 销售查询

主服务启动时会连接：

rag:
  mcp:
    servers:
      - name: default
        url: http://localhost:9099

启动日志里如果看到：

MCP Server [default] 返回 3 个工具

说明工具注册成功。

MCP 调用流程#

用户问题
  -> 意图识别发现这是工具型问题
  -> LLM 提取工具参数
  -> RemoteMCPToolExecutor 调用 MCP Server
  -> MCP Server 执行工具
  -> 工具结果返回主服务
  -> 主服务把工具结果拼进 Prompt
  -> LLM 组织自然语言答案

核心类：

  • LLMMCPParameterExtractor
  • DefaultMCPToolRegistry
  • RemoteMCPToolExecutor
  • HttpMCPClient
  • MCPClientAutoConfiguration

RAG 基础概念 - 对话增强篇#

为什么需要对话增强#

真实用户不会每次都问完整问题。

第一轮：

公司年假怎么申请？

第二轮：

那需要提前几天？

第二轮里的“那”依赖第一轮。如果不加载会话历史，系统不知道用户在问年假。

Ragent 的会话记忆#

核心类：

类 作用
ConversationMemoryService 会话记忆接口
DefaultConversationMemoryService 默认实现
JdbcConversationMemoryStore 从数据库读取和保存消息
ConversationMemorySummaryService 会话摘要

配置：

rag:
  memory:
    history-keep-turns: 4
    summary-start-turns: 5
    summary-enabled: true
    summary-max-chars: 200

意思是：

  • 近期 4 轮直接保留。
  • 长对话开始摘要。
  • 摘要控制长度，减少 token。

问题改写与拆分#

核心类：

QueryRewriteService
MultiQuestionRewriteService
RewriteResult

例子：

用户问：

报销和请假分别怎么走？

系统可以拆成：

1. 报销流程是什么？
2. 请假流程是什么？

这样每个子问题可以单独识别意图和检索，提高召回率。

模糊意图澄清#

如果用户问：

怎么申请？

系统可能无法判断是申请年假、报销、账号权限，直接回答容易错。

Ragent 里有：

IntentGuidanceService
GuidanceDecision

它会判断是否需要引导用户补充信息。

RAG 基础概念 - 数据传输协议#

为什么问答用 SSE#

大模型生成答案需要时间。如果等完整答案生成后再返回，用户会感觉卡住。

SSE 可以让后端不断把片段推给前端：

模型生成一点 -> 后端发送一点 -> 前端渲染一点

后端接口：

GET /api/ragent/rag/v3/chat

控制器：

RAGChatController

输出类型：

text/event-stream;charset=UTF-8

SSE 和 WebSocket 的区别#

对比 SSE WebSocket
通信方向 服务端到客户端为主 双向
协议复杂度 简单 较复杂
断线重连 浏览器原生支持 需要自己处理
适合场景 大模型流式输出、通知 实时协作、游戏、双向聊天

Ragent 的问答流是服务端持续推送，所以 SSE 足够。

前端如何接收#

前端相关代码：

frontend/src/hooks/useStreamResponse.ts
frontend/src/services/chatService.ts
frontend/src/components/chat/MessageList.tsx
frontend/src/components/chat/MessageItem.tsx

学习时重点看：

  • 前端如何打开 SSE 连接。
  • 如何解析不同事件类型。
  • 如何把增量内容追加到消息里。
  • 用户点击停止时如何调用后端停止接口。

RAG 基础概念 - 模型检索生成评估与优化#

RAG 效果差应该查哪里#

不要只盯着最终回答。要按链路排查：

问题 排查点
答非所问 看问题改写是否正确
检索不到 看知识库是否入库成功
检索结果不准 看 chunk 粒度和向量相似度
答案引用错 看 Rerank 和 Prompt
多轮错乱 看会话记忆和摘要
工具没调用 看意图树和 MCP 工具注册
模型报错 看模型 Key、供应商响应和路由降级

Ragent 的 Trace 能看什么#

相关表：

t_rag_trace_run
t_rag_trace_node

相关类：

RagTraceAspect
RagTraceContext
RagTraceRecordService
RagTraceQueryService

前端页面：

frontend/src/pages/admin/traces

Trace 应该关注：

  • 每个节点耗时。
  • 输入输出摘要。
  • 是否有异常。
  • 检索命中的 chunk。
  • 最终 Prompt 是否合理。

检索优化方法#

优化项 说明
分块策略 结构化文档优先用结构感知分块
topK 太小漏召回，太大噪音多
Rerank 用重排模型把最相关 chunk 提前
意图树 缩小检索范围，提高准确度
术语映射 把口语表达映射到业务词
元数据过滤 限定知识库、文档类型、业务域

生成优化方法#

优化项 说明
Prompt 边界 要求基于上下文回答
答案结构 让模型按步骤、表格或条目输出
上下文排序 高相关内容放前面
空检索处理 没查到就提示，不要硬答
temperature 知识问答建议低温度
